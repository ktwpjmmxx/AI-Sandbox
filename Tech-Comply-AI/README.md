## 📖 概要

**Tech-Comply AI** は、Webサービス開発やマーケティング業務において発生する「法的リスク」を開発段階で早期に検知し、修正を支援する文書診断システムです。

### 解決する課題

Webサービス開発・運営の現場では、以下の課題がボトルネックとなっています：

| 課題 | 影響 | Tech-Comply AIの解決策 |
|------|------|---------------------|
| **コンプライアンスリスク** | 広告表現や利用規約の不備による炎上・法的制裁 | 開発段階での自動リスク検知 |
| **確認コスト** | 些細な文言確認でも法務部への相談が必要 | 一次チェックの自動化で工数削減 |
| **専門知識不足** | 開発者が法律の詳細を把握しきれない | 条文・判例に基づく根拠付き回答 |

### 汎用LLMとの違い

一般的なLLM（ChatGPT、Claude等）との差別化ポイント：

- ✅ **ハルシネーション（誤情報）の抑制** - 日本の法令データでファインチューニング
- ✅ **根拠の明示（グラウンディング）** - 必ず条文番号・法律名を引用
- ✅ **構造化された出力** - リスクレベル、該当法、理由、修正案を一貫したフォーマットで提示

## 🎯 主な機能

### 1. 根拠付きリスク判定（Grounding）

入力されたテキスト（広告文、利用規約、契約条項）に対し、関連する法律の**条文番号**を引用して回答します。

**出力例**:
```
リスクレベル：高
該当法：景品表示法第5条第1号（優良誤認表示の禁止）
理由：「業界No.1」という表現は、客観的な根拠なく優位性を示唆する...
修正案：「〇〇調べ（調査期間：2024年1-3月、n=500）業界トップクラス」のように...
```

### 2. リスクスコアリング

コンプライアンス違反のリスクを**「高・中・低」の3段階**で判定し、優先的に修正すべき箇所を提示します。

### 3. 実務で使える修正提案

法的要件を満たすための具体的な修正案を、実務ですぐに使える形で提供します。

## 🔬 技術スタック

### モデルアーキテクチャ

```
ベースモデル: ELYZA-japanese-Llama-2-7b
ファインチューニング: QLoRA (4-bit量子化)
学習手法: Supervised Fine-Tuning (SFT)
フレームワーク: PyTorch, Hugging Face Transformers, PEFT
```

### ファインチューニング詳細

| 項目 | 設定値 | 説明 |
|------|--------|------|
| **LoRAランク** | 16 | 過学習を防ぎつつ表現力を確保 |
| **学習率** | 2e-4 | 安定した学習のための最適値 |
| **エポック数** | 5 | 十分な学習と過学習防止のバランス |
| **データセット** | 150件（修復済み） | 景品表示法、個人情報保護法等 |

### 対応法令

- 景品表示法（不当表示の禁止）
- 個人情報保護法（取得・利用・提供）
- 特定商取引法（通信販売、サブスク規制）
- 著作権法（引用、二次利用）
- フリーランス保護新法（発注・契約）

## 🚀 クイックスタート

### 前提条件

- Python 3.10以上
- GPU環境（T4以上推奨、Google Colab可）
- 必要パッケージ：`transformers`, `peft`, `trl`, `bitsandbytes`

### 実行手順

#### Step 1: データ修復（初回のみ）

```bash
# 学習データの修正案フィールドを修復
python scripts/fix_training_data.py \
  data/train_data_natural_language.jsonl \
  data/train_data_fixed.jsonl
```

#### Step 2: ファインチューニング実行

```bash
# v4設定で学習（6-10時間 on T4）
python scripts/train_finetuning_v4.py \
  --config_path config/config_v4.yaml
```

#### Step 3: 推論テスト

```bash
# 学習済みモデルで診断実行
python scripts/inference_v4.py \
  --adapter_path ./results/results_v4/final_adapter \
  --config_path config/config_v4.yaml
```

詳細な手順は [v4ドキュメント](./docs/README_V4.md) を参照してください。

## 🔧 技術的ハイライト

### 1. データ品質の徹底管理

**発見した問題**: 150件中50件（33%）で修正案フィールドが空白

**解決策**: `fix_training_data.py`で自動補完ツールを開発
- リスクレベルに応じた汎用修正案を生成
- 修復率100%達成

### 2. システムメッセージとデータの完全一致

**問題**: 学習データは「リスクレベル：」なのに、システムメッセージは「【リスクレベル】」

**解決**: フォーマットを完全統一し、モデルの混乱を解消

### 3. 推論パラメータの最適化

| パラメータ | v3（問題あり） | v4（最適化） | 根拠 |
|-----------|--------------|-------------|------|
| temperature | 0.1 | **0.6** | 法務AIの業界標準 |
| LoRAランク | 32 | **16** | 過学習防止 |
| 評価頻度 | epoch単位 | **50 steps** | 細かく監視 |

## 🎓 開発の学びと考察

### 成功要因

1. **データ品質が全て** - 量より質。150件でも欠損があれば機能しない
2. **学習と推論の一貫性** - プロンプトフォーマットの統一が最重要
3. **適切なパラメータ設定** - temperature 0.1は法務用途では低すぎる

### 失敗から学んだこと

- v2: データ量を増やしても品質が低ければ効果薄
- v3: JSON特化を試みるも、根本原因（データ欠損）を見落とす
- v4: 原因分析を徹底し、体系的に解決

詳細は各バージョンのREADMEを参照してください。
