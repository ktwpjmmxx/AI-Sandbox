## v3の実測結果と問題点

### v3で発生した問題

```
リスクレベル：中〜高
該当法：割賦販売法、消費者契約法
理由：単なる分割払いではなく、利用者が商品を受領した時点で代金の一部を支払うことを前提とした決済方法である場合、割賦販売に該当します。特に、返品やキャンセルが発生しやすい商品を対象に、そのリスクを回避するために設定された決済方法である場合、割賦販売の範囲内に含まれます。
例外として、借り入れ型ローンの審査基準を満たしている場合は、割賦販売ではなく借り入れとみなされ、対象外となります。
修正案：割賦販売の範囲内であることを明示し、審査基準を満たしていることを客観的に証明できる仕様に変更しましょう。また、契約書面や注意事項等に、割賦販売の特性を十分に伝える必要があります。
注意点：割賦販売の範囲内であっても、審査基準を満たしていない場合は、割賦販売
----------------------------------------------------------------------
                                                                      
以上
```

#### **致命的な問題**

| # | 問題 | 影響 |
|---|------|------|
| 1 | **途中で出力が途切れる**<br>「修正案：...割賦販売」で終了 | 実用不可 |
| 2 | **フォーマットが不安定**<br>「中〜高」「【】なし」等ブレる | 品質低下 |
| 3 | **同じ表現の繰り返し**<br>「割賦販売」が5回以上登場 | 冗長 |

---

## 原因分析

### 1. **学習データの致命的欠陥（最重要）**

```json
// 150件中50件（33%）がこの状態
{"output": "リスクレベル：高\n該当法：...\n理由：...\n修正案："}
                                                              ↑ 空白！
```

**影響**: モデルが「修正案は書かなくていい」と学習してしまう

### 2. **システムメッセージとデータ形式の不一致**

| 項目 | 学習データ | システムメッセージ(v3) |
|------|-----------|---------------------|
| フォーマット | `リスクレベル：高` | `【リスクレベル】` |
| 区切り | `：`（全角） | `【】`（括弧） |

**影響**: モデルが混乱し、安定した出力ができない

### 3. **推論パラメータの不適切な設定**

```python
temperature=0.1  # ← 極端に低すぎる
```

- **0.1の問題**: ほぼ決定的な出力になり、同じフレーズを繰り返す
- **法務文書の適正値**: 0.5〜0.7（正確性と自然さのバランス）

### 4. **LoRAランク32による過学習**

150件のデータに対してランク32は大きすぎ → 訓練データを丸暗記

## v4での改善策

### 改善1: 学習データの自動修復

```python
# fix_training_data.py を実行
python fix_training_data.py train_data_natural_language.jsonl train_data_fixed.jsonl
```

**実施内容**:
- 空白の「修正案：」を検出（50件）
- リスクレベルに応じた汎用修正案を自動生成
- 修復率: **100%**

### 改善2: システムメッセージの統一

**Before (v3)**:
```
【リスクレベル】
【該当法】
【法的根拠・理由】
【修正案】
```

**After (v4)**:
```
リスクレベル：
該当法：
理由：
修正案：
```

→ 学習データと完全一致

### 改善3: 推論パラメータの最適化

| パラメータ | v3 | v4 | 根拠 |
|-----------|----|----|------|
| temperature | 0.1 | **0.6** | 法務AIの業界標準 |
| top_p | 0.9 | **0.85** | やや保守的に |
| top_k | なし | **40** | 候補を絞る |
| repetition_penalty | 1.1 | **1.15** | 繰り返し抑制強化 |

### 改善4: LoRAランクの削減

```yaml
peft_config:
  r: 16              # 32 → 16（過学習防止）
  lora_alpha: 32     # 64 → 32（ランクの2倍）
  lora_dropout: 0.1  # 0.05 → 0.1（汎化性能向上）
```

### 改善5: 評価頻度の向上

```yaml
training_arguments:
  save_strategy: "steps"    # epoch → steps
  save_steps: 50
  eval_strategy: "steps"    # epoch → steps  
  eval_steps: 50
```

→ 細かく進捗を監視し、過学習を早期発見

## トラブルシューティング

### 問題1: まだ出力が途切れる

**確認事項**:
```bash
# データ修復が完了しているか確認
grep '修正案：$' train_data_fixed.jsonl | wc -l
# → 0件であることを確認
```

**対処法**:
- エポック数を7に増やす
- `max_new_tokens`を768に増やす

### 問題2: フォーマットがブレる

**原因**: システムメッセージとデータの不一致

**対処法**:
```python
# inference_v4.py の system_message を確認
# 「リスクレベル：」となっているか確認（【】なし）
```

### 問題3: 同じ表現の繰り返し

**対処法**:
```yaml
# config_v4.yaml を編集
inference:
  repetition_penalty: 1.2  # 1.15 → 1.2
  temperature: 0.7         # 0.6 → 0.7
```
