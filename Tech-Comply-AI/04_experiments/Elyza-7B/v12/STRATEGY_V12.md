# Elyza-7B v12 改善戦略書

## エグゼクティブサマリー

v12は「**知識の強制上書き**」を主たる目的とし、v11で残存した拒絶バイアス問題を解消することを目指す。学習強度を段階的に引き上げ、ベースモデルに刻まれた「法的断定を避ける傾向」をデトックス済みデータで塗り替える。

---

## v11の詳細分析

### 成功した点
1. **学習の安定性**: Loss 0.64、grad_norm 0.7-0.8で非常に安定
2. **構造化出力の維持**: 4項目フォーマット（リスクレベル/該当法/理由/修正案）が崩れていない
3. **デトックスの方向性**: データクリーニングのアプローチは正しい

### 失敗した点
1. **知識の活用失敗**: 学習データ内の質問（Googleアナリティクス）に「専門外」と回答
2. **学習強度の不足**: lr=1e-4、2 epoch では拒絶バイアスを上書きできなかった
3. **提案の未実行**: strategy_doc.mdで提案された「lr=2e-4、3 epoch、alpha=128」を実行できていない

## v12の改善戦略

### コア戦略: 知識の強制上書き

**方針**: 拒絶バイアス→新しい知識で上書きする

### パラメータ調整の理論的根拠

| パラメータ | v11 | v12 | 変更理由 |
|-----------|-----|-----|----------|
| **num_train_epochs** | 2.0 | **3.0** | v10で3.0が機能。接触回数を増やし拒絶パターンを上書き |
| **learning_rate** | 1e-4 | **2e-4** | v10と同じ強度。知識定着を優先（ただし過学習リスクに注意） |
| **lora_r** | 32 | **32** | 維持。これ以上上げると過学習、下げると表現力不足 |
| **lora_alpha** | 64 | **128** | r:alpha = 1:4。学習データへの追従性を最大化 |
| **lora_dropout** | 0.05 | **0.0** | 迷いを完全排除。デトックス済みデータを直接反映 |
| **warmup_steps** | 10 | **5** | 早期から本格学習開始。ウォームアップは最小限 |
| **lr_scheduler** | cosine | **linear** | v10で機能していたシンプルな線形減衰 |

### 設定の数学的解釈

#### LoRA Alpha の役割
```
effective_learning_rate = learning_rate * (lora_alpha / lora_r)
v11: 1e-4 * (64 / 32) = 2e-4
v12: 2e-4 * (128 / 32) = 8e-4
```
→ v12は実効学習率で **4倍** の強度

#### Dropout 0 の意味
- `dropout > 0`: モデルが汎化を学ぶ（多様なパターンに対応）
- `dropout = 0`: モデルがデータを直接記憶（特定パターンに特化）

v12では後者を選択。デトックス済みデータの質が高いため、「迷わず学習せよ」という指示。

リスクレベル：中

## リスク分析と対策

### リスク1: 過学習（Overfitting）

**懸念点**:
- 3 epoch + lr 2e-4 + alpha 128 は強力すぎる可能性
- 特定フレーズの暗記に陥る危険

**対策**:
1. **エポックごとのチェックポイント保存**: `save_strategy: "epoch"`で各エポックの状態を保存
2. **段階的評価**: Epoch 1, 2, 3 それぞれで推論テストを実施
3. **Early Stopping の検討**: Epoch 2.5で既に良好なら、3.0まで回す必要なし

### リスク2: 構造化出力の崩壊

**懸念点**:
- 強い学習圧力により、フォーマットが乱れる可能性

**対策**:
- データセット内の全サンプルで「リスクレベル」「該当法」「理由」「修正案」の4項目が統一されているか再確認
- もし崩壊した場合は Epoch 2 のチェックポイントにロールバック

### リスク3: ハルシネーションの再発

**懸念点**:
- v10で克服したハルシネーションが、強力な学習により復活する可能性

**対策**:
- 推論テストで「存在しない法令」「虚偽の判例」を語っていないか厳密にチェック
- もし発生した場合、データセット内の「創作的な例」を削除して再学習

## v13 への展望

v12の結果に応じて、以下の方向性を検討:

### シナリオA: v12が成功した場合
- **目標**: 安定性と品質の向上
- **調整**: lr を 1.5e-4 に下げ、過学習を抑制
- **追加**: 検証データセット（validation split）の導入

### シナリオB: v12が失敗した場合
- **目標**: データセットの根本的見直し
- **調整**: 「拒絶データ」を完全削除し、ポジティブデータのみで学習
- **追加**: データ拡張（GAの質問を5パターンに増やす）

### シナリオC: v12で過学習が発生した場合
- **目標**: 学習強度の適正化
- **調整**: Epoch 2.0、lr 1.5e-4、alpha 96 に調整
- **追加**: Validation Loss によるEarly Stopping

## 成功の定義

v12は以下の条件を満たせば成功とみなします:

1. **必須条件**:
   - Googleアナリティクス質問に対し、「電気通信事業法」「外部送信規律」に言及する
   - 「専門外」ではなく、具体的なリスクと対策を提示する

2. **望ましい条件**:
   - 4項目フォーマットが維持されている
   - 他のIT法務質問（Cookie、利用規約等）にも適切に回答
   - 技術的質問には引き続き「専門外」と適切に拒絶

3. **絶対に避けるべき事態**:
   - ハルシネーション（存在しない法令の創作）
   - フォーマット崩壊（箇条書きのみ、冗長すぎる回答等）
   - 全質問への無差別回答（専門外の概念消失）

## まとめ

v12では「慎重すぎるコンサルタント」から「実務的なアドバイザー」に変える実験。

**キーメッセージ**:
> "We're not weakening the guardrails.  
> We're teaching the model where the boundaries are."

v12では対応範囲の境界線を学習を通してモデルにしっかりと覚えさせる。
対応範囲外の質問に対しても単純に拒絶するのではなく、対応できない理由を出力内に添えられるのがベスト。
