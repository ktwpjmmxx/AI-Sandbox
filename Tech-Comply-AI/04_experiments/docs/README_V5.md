## 技術的課題と解決：Unslothとの闘い

### 直面した致命的エラー

Exception: Unsloth: Critical error since some weights are not initialized.
Please try updating Unsloth, transformers and timm...

#### 原因分析
ELYZA（Llama-2ベース）をロードする際、内部パラメータ（inv_freq等）の微細な不一致が発生。通常のライブラリでは「警告」で済むところを、Unslothの厳格なガードレール仕様により「致命的エラー」としてプログラムが即時強制終了される事態が発生しました。

#### 解決策（エンジニアリング・ハック）
1. ログレベルの強制引き上げ: transformersのログ出力閾値をERRORに設定し、警告を非表示化。これにより、Unsloth側の「警告検知→強制終了」のトリガーを回避。
2. 互換性フラグの適用: ignore_mismatched_sizes=True をモデルロード時に適用し、パラメータの不一致を許容。
3. 推論時パッチ: 推論スクリプトにも同様のログ制御を導入し、アダプターロード時のクラッシュを完全に抑制。

## v5での構成・設定詳細

### 改善1: Unslothによる高速化設定

モデルロード時にUnslothのFastLanguageModelを使用し、4-bit量子化とエラー回避フラグを同時に適用。

- model_name: config["model_name"]
- max_seq_length: 1024
- load_in_4bit: True
- ignore_mismatched_sizes: True

### 改善2: システムメッセージとプロンプトの完全統合

学習データと推論時のプロンプトを以下の形式で完全統一。

プロンプト構造:
<s>[INST] <<SYS>>
あなたはIT法務およびコンプライアンスの専門コンサルタントです。
<</SYS>>
{instruction}
{input} [/INST] {output} </s>

### 改善3: 学習パラメータの最適化

| パラメータ | v4 | v5 | 根拠 |
|-----------|----|----|------|
| **エンジン** | Transformers | **Unsloth** | 高速化とVRAM節約の両立 |
| **Loss (最終)** | 0.85 | **0.69** | 学習の収束性が大幅に向上 |
| **Learning Rate** | 2e-4 | **1e-4** | 350件への拡張を見据えた安定重視 |
| **Optimizer** | AdamW | **adamw_8bit** | 8bit量子化によるメモリ最適化 |
