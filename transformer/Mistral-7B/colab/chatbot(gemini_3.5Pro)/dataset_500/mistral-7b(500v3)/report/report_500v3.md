## 実験結果（Exp-002）

### モデルに現れた主な症状

#### 1. トークンループ（#記号の繰り返し）
**症状**: 応答の途中で「####」などの無意味な記号列が出力され、生成が停止する。

**具体例**:
- 入力: 「『ありがとう』と言われたら、あなたはどのように返答するように学習されましたか？」
- 実際の出力: `学  #  #  #  #  #  #  #  #...（延々と続く）`

#### 2. 完全に無関係な応答
**具体例**:
```
Q: 地球の地軸の傾きがもし20度ではなく45度だった場合、季節の変化に具体的にどのような影響が出ると考えられますか？
A: 地球は水の沸点ですので、壁に触れると熱血だといいですよ。
```
→ 完全に意味不明で、学習データの断片を無作為に組み合わせている

#### 3. 特定フレーズの過度な暗記
**頻出する暗記フレーズ**:
- 「諦めるな！お前ならできる！壁なんてぶち壊して進めばいいんだ！全力でサポートするぞ！」
- このフレーズが文脈無視で複数回出現

#### 4. 短すぎる/空白の応答
- 「はい、」だけで終わる応答が複数
- 完全に空白の応答もある
- 質問の複雑さに応じた応答ができていない

### 過学習の決定的証拠

#### 1. Lossの異常な低下
```
Epoch 1.0:  Loss = 0.4085
Epoch 2.0:  Loss = 0.1881 (54%減少)
Epoch 3.0:  Loss = 0.1094 (73%減少 from Epoch 1)
```
→ **検証データなし**でこの数値は危険信号

#### 2. Entropyの急激な低下
```
Epoch 0.03: Entropy = 2.4112
Epoch 1.0:  Entropy = 0.4831 (80%減少)
Epoch 3.0:  Entropy = 0.2918 (88%減少)
```
→ **モデルが多様な出力を生成できなくなっている**

#### 3. Token Accuracyの早期飽和
- Epoch 1.0の時点で既に94.5%
- これ以上の学習は過学習を深めるだけ

#### 4. Gradient Normの不安定さ
```
Step 67: grad_norm = 16.7145 (異常なスパイク)
Step 76: grad_norm = 4.6168
Step 82: grad_norm = 6.0726
```
→ 学習が不安定になっている証拠

## 根本原因の分析（Exp-002）

### 主要な原因

#### 1. **検証データの完全欠如**（最重要）
- Training Lossしか監視していない
- Validation Lossがないため、過学習を検出できなかった
- Early Stoppingが不可能

#### 2. **過剰な学習回数**
- **3 epoch は多すぎた**
- Epoch 1の時点でLoss = 0.4085、Accuracy = 94.5%
- Epoch 2-3は完全に過学習を深めただけ

#### 3. **学習率スケジュールの問題**
- Cosine annealingを使用
- 学習率が徐々に下がり、細かいパターンまで暗記
- 最終的にLR = 2.13e-06まで低下

#### 4. **データの多様性不足**
- 500件のデータに対して3 epoch = 1,500回の反復
- 同じパターンを何度も見せすぎた
- 特に「熱血教師」のような特徴的なパターンを過剰に学習

#### 5. **正則化の不足**
- LoRAを使用しているが、それだけでは不十分
- Dropoutやweight decayの設定が確認できない

## 過学習の具体的な症状とメカニズム

### なぜ「#」が繰り返されるのか？
1. 特定のトークンパターンを暗記
2. コンテキストを理解せず、暗記したパターンをループ
3. EOS（文章終了）トークンを適切に生成できない

### なぜ無関係な応答が出るのか？
1. 質問の意味を理解していない
2. 学習データの断片（「水の沸点」「熱血」）を無作為に組み合わせ
3. 学習データに類似した単語があると、そのデータ全体を想起

### なぜ同じフレーズが頻出するのか？
1. 「熱血教師」のデータが印象的だった
2. 高いLossが出た部分を過剰に学習
3. そのパターンが他の文脈でも出現するようになった

## 傾向と対策

### 今回の実験で学んだこと（Exp-002）

#### うまくいかなかったこと
1. **検証データなしで3 epochも学習した**
   - 過学習の検出が不可能だった
   
2. **Loss値だけで判断した**
   - Entropyの低下を見逃した
   - Token Accuracyの早期飽和を軽視した

3. **データ量を増やせば良いと考えた**
   - 500件でも3 epochは多すぎた
   - 量より質と学習回数のバランスが重要

#### 気づいたポイント
1. **Epoch 1の時点で既に十分な性能が出ていた**
   - Loss = 0.4085, Accuracy = 94.5%
   - ここで止めるべきだった

2. **Entropyは重要な指標**
   - 0.4以下になったら危険信号
   - 多様性の喪失を示している

3. **学習ログの可視化が必須**
   - 数値だけ見ていても気づきにくい
   - グラフ化すれば一目瞭然

## まとめ

### Key Takeaways（更新版）

1. **検証データは絶対に必要** ✓
   - Exp-005で完全実装
   - Training Lossだけでは過学習を検出不可能
   - 最低でも15%は検証用に確保すべき（Exp-005で実装）

2. **過学習は防げた** ✓
   - Exp-002: 3 epochs で過学習
   - Exp-005: 2 epochs で安定
   - 適切なEpoch数の重要性を確認

3. **Entropyは重要な警告サイン** ✓
   - Exp-002: 0.29まで低下（危険）
   - Exp-005: 0.60を維持（健全）
   - 多様性の喪失を早期に検出できる

4. **Early Stoppingは必須機能** ✓
   - Exp-005で実装済み
   - 人間の判断に頼らず、自動的に停止させる仕組みが機能

5. **技術指標と実用性は別物** ⚠️ **新規発見**
   - 過学習は防げても、日本語品質が悪化
   - Loss/Accuracyだけでなく、実際の出力品質も評価すべき
   - **ベースモデルの選択が最重要**

6. **プロンプトエンジニアリングの限界** ⚠️ **新規発見**
